MyWechatBot 核心代码逻辑解读

这份文档按“消息如何进入、如何判断、如何调用模型和工具、如何记忆、如何回复”的路径解释当前代码。重点覆盖机器人逻辑，不展开依赖安装和部署细节。

总览 文件职责 消息主链路 路由与 Agent 记忆与存储 多模态 状态页 Root 权限 回复处理 设计边界

1. 总体架构

当前项目可以理解为四层：微信接入层、机器人决策层、持久化状态层、模型/工具调用层。

入口Wechaty

src/main.ts 负责扫码、登录、监听消息，并把消息交给 WechatCompanion。

核心Companion

src/companion.ts 是主脑，处理群聊模式、路由、提醒、文件、代码执行、多模态、回复。

状态BotStore

src/store.ts 使用 JSON 文件保存聊天记录、记忆、提醒、token 用量、root、白名单。

模型OpenAI 兼容 API

Config.openaiBasePath 指向 DashScope 兼容接口，聊天、agent、视觉、语音模型可分开配置。

微信消息
  -> src/main.ts
  -> WechatCompanion.onCustimzedTask()
  -> WechatCompanion.onMessage()
  -> 路由判断 / 工具执行 / 普通聊天
  -> BotStore 记录状态与用量
  -> botStatus 更新运行状态
  -> 分段清洗后发回微信

2. 核心文件职责

文件 职责 你改功能时常看哪里
src/main.ts 创建 Wechaty 实例；打印二维码；登录后启动机器人；监听消息；过滤重启前历史消息。 扫码、登录、服务启动、消息入口。
src/config.ts 优先读取 config.yaml，没有则读取环境变量；设置默认模型、群聊模式、冷却、回复长度、多模态等。 新增配置项、修改默认值、排查配置是否生效。
src/interface.ts 定义配置接口 IConfig。 新增配置时同步类型。
src/status-server.ts 维护只读运行状态快照，提供 HTML 状态页和 /status.json。 查看模型是否生效、微信是否登录、最近 API/错误摘要。
src/store.ts 封装 JSON 存储，负责消息、长期记忆、提醒、token 记录、root、白名单、群设置。 记忆隔离、历史记录、提醒状态、统计逻辑。
src/companion.ts 最大核心文件，包含消息处理、工具路由、Agent 任务、多模态、root 管理、回复清洗和分段。 绝大多数机器人行为都在这里。

3. 消息主链路

一条消息从微信进来后，不会马上扔给大模型，而是经过一串前置判断。

1

main.ts 收到消息

启动时先开启可选状态页；收到消息后检查消息时间，避免服务重启后回复历史消息。然后调用 onCustimzedTask()，如果没有被处理，再进入 onMessage()。

2

自定义命令优先处理

onCustimzedTask() 先处理公众号忽略、root 授权、root 命令、群聊模式切换、帮助菜单等。这些不需要大模型。

3

创建会话上下文

createContext() 判断消息来自私聊还是群聊，记录 chatId、chatName、talkerId、talkerName。这是隔离记忆和权限的基础。

4

过滤与权限判断

isIgnorable() 忽略公众号、系统消息、红包、撤回、部分无法处理类型；isChatAllowed() 根据 root 设置的白名单/黑名单决定是否响应。

5

非文本消息进入多模态链路

图片、语音、视频、表情包由 handleNonTextMessage() 处理。它会先缓存最近媒体，再决定是否主动调用视觉/语音模型。

6

文本消息进入响应判断

shouldRespond() 根据私聊自动回复、群聊模式、@、触发词、是否需要工具、冷却时间决定要不要说话。

7

路由到聊天或工具

handleUserText() 先做确定性识别，再交给 LLM 路由兜底，最后由 handleIntent() 执行具体动作。

4. 路由与 Agent 逻辑

当前采用“双层路由”：先用规则抓明显需求，再让便宜的 agent 模型判断模糊自然语言，避免纯关键词误伤。

确定性路由

  • parseDeterministicToolIntent() 识别 token 用量、提醒、提醒列表、文件生成、代码运行、日程表等。
  • 优点是快、便宜、稳定，很多需求无需多一次 LLM 调用。
  • 风险是自然语言里嵌入关键词可能误判，所以后面有 guard。

LLM 路由

  • routeIntentIfNeeded() 使用 Config.agentModel 输出 JSON 意图。
  • 支持 remember、recall、reminder、agent_task、usage_report、set_group_mode 等。
  • 提示词明确要求：不要把书名或普通话题里的 token 误判成用量查询。
关键设计：guardDeterministicIntent() 会把高风险或容易误伤的确定性意图再交给 LLM 审一遍。例如“请搜索《token 的一生》”不应该触发 token 用量查询。
意图 执行函数 效果
chat onCompanionChat() 普通聊天，带最近消息和相关记忆。
remember handleRemember() 保存长期记忆到当前会话。
recall handleRecall() 检索当前会话记忆并自然回答。
reminder handleReminder() 创建一次性、每日、连续多次提醒。
agent_task handleAgentTask() 生成文件、日程表、计划，或在确认后执行代码。
usage_report handleUsageReport() 查询当前会话 token 用量，root 可看全局导出。

5. 记忆、历史与数据隔离

BotStore 的核心原则是：聊天记录可以一直保存，但每次喂给模型的上下文只取最近一小段，再加检索到的相关长期记忆。

数据 存储字段 隔离方式
原始聊天记录 messages 每条都有 chatId 和 talkerId，查询最近上下文时按 chatId 过滤。
长期记忆 memories searchMemories() 当前只搜索本会话 chatId 的记忆，因此不同用户/群隔离。
提醒 reminders 提醒绑定创建时的会话，群聊提醒发到对应群，私聊提醒发给对应联系人。
用量记录 usageRecords 按 date、chatId、scope 汇总，root 或配置允许时可看全局。
root 与白名单 rootUsers、chatAccess root 绑定 talkerId；白名单/黑名单绑定会话 chatId。 
普通聊天上下文 = [
  system 人格与规则,
  当前会话信息,
  当前会话相关长期记忆,
  最近 N 条同 chatId 消息,
  用户本次消息
]
注意：当前“记忆检索”是简单关键词打分，不是向量库。好处是轻量和可控；缺点是语义召回能力有限，后续可以升级成 embedding + SQLite/向量索引。

6. 多模态逻辑

多模态不是每条消息都调用模型，而是先缓存、再判断是否处理，以控制 token 和避免刷屏。

图片

通过 describeImageMessage() 调用视觉模型，提示词要求识别图片内容、文字、截图细节。

表情包

describeEmoticonMessage() 把表情包转成一句语义短语，并用 emoticonMeaningCache 缓存。

语音

transcribeAudioMessage() 下载微信语音并转成 wav，再用 qwen3-omni-flash 的流式接口转写。

视频

describeVideoMessage() 用 ffmpeg 抽帧，再交给视觉模型总结视频内容。

最近媒体引用：图片/语音/表情/视频会进入 pendingMediaByChat，10 分钟内用户说“看一下这张图”“这个表情什么意思”时，会关联最近媒体。
主动处理策略：私聊默认可主动处理图片/语音/表情；群聊在安静和智能模式下不主动看媒体，超活跃/话唠模式才更积极，且有冷却。
语音转写细节：普通聊天仍走 completeChat()，但语音转写走 completeOmniAudioTranscription()。原因是 Qwen-Omni 要求音频 base64 使用 data:;base64,... 包装，并且请求必须 stream: true；否则会出现“URL 不合法”类 400 错误。

7. 运行状态页

状态页用于快速确认机器人是否在线、模型配置是否生效、最近 API 调用和错误摘要是什么。它不展示聊天原文、二维码、API Key 或 root 授权串。

启动方式

  • Config.statusPageEnabled 控制是否启用。
  • statusHost 默认绑定 127.0.0.1，避免直接暴露端口。
  • 服务器通过 Nginx 将 /wechat-status/ 反代到状态服务。

记录内容

  • 微信状态：启动中、等待扫码、已登录、已退出、异常。
  • 模型快照：聊天、agent、视觉、语音四个模型名。
  • 计数摘要：收发消息数、API 调用数、token 总量、最近错误首行。
公开入口：
http://dawnfy.top/wechat-status/
http://dawnfy.top/wechat-status/status.json

8. Root、白名单与群聊模式

Root 是最高权限用户，用来管理机器人可以在哪些会话里说话，以及查看会话列表和摘要。

能力 入口 代码逻辑
获取 root 发送配置里的 rootAuthToken 或 /root token isRootAuthText() 校验，成功后 store.addRootUser()。
查看好友/群/会话 /好友列表、/群列表、/会话列表 优先从 Wechaty 实时读取，失败时从 knownChats 兜底。
禁用或允许会话 /禁止 编号、/允许 编号 setChatAccess() 写入 chatAccess，后续 isChatAllowed() 判断。
总结其他会话 /总结 编号 读取目标会话最近 160 条消息，用 agent 模型生成摘要。

群聊模式

模式 行为 适合场景
quiet 只在被 @ 或触发词出现时回复。 大群、工作群、怕打扰的群。
smart 明显问到机器人或需要工具时回复。 默认模式，比较稳。
active 更主动参与，有冷却。 熟人小群。
super_active 更积极接话，媒体处理也更主动。 测试群、娱乐群。
talkative 高频聊天，语音等也更可能处理。 想把它当群友玩的时候。

9. 模型调用、Token 记录与回复前处理

大多数聊天、路由、视觉理解、摘要等模型调用经过 completeChat()；语音转写因为 Qwen-Omni 的接口要求，单独走流式请求。

  • completeChat() 调 OpenAI 兼容接口，默认使用 Config.openaiModel，路由和部分 agent 使用 Config.agentModel。
  • completeOmniAudioTranscription() 使用 Config.audioModel，以流式方式读取转写文本和 usage。
  • 如果 API 返回 usage，就记录真实 token；如果没有返回，就用字符长度粗估。
  • store.addUsage() 会保存 feature、model、chatId、scope、promptTokens、completionTokens、totalTokens。
  • botStatus.recordUsage() 同步更新状态页里的 API 调用数、最近模型和 token 总数。
  • prepareReplyForWechat() 会去掉 Markdown 粗体、代码围栏、链接格式等，避免微信里出现一堆星号。
  • splitWechatMessage() 按最大长度切分消息，优先在换行和标点处断开，超过最大段数会截断并提示。
文件发送：生成日程表、报告、token 用量文件时，先写入 Config.generatedFilesPath，再通过 pendingFilesByChat 在回复后发送文件。

10. 当前设计优点与边界

已经做得不错的地方

  • 私聊、群聊、root、白名单、群模式已经有明确边界。
  • 提醒、每日提醒、连续提醒、多模态、文件生成都有基础实现。
  • token 统计是系统真实记录，不再依赖模型编造。
  • 公网状态页能快速确认模型配置和运行状态，排查时不用一直翻日志。
  • 代码执行有确认机制，避免看到“代码”两个字就执行。
  • 媒体处理有缓存和冷却，避免表情刷屏烧 token。

后续最值得优化的地方

  • 把 companion.ts 拆成路由、工具、媒体、回复、权限等模块，降低维护难度。
  • 把 JSON 存储升级为 SQLite，方便查询、索引、备份和迁移。
  • 长期记忆增加 embedding 检索和定期摘要，解决历史越来越多的问题。
  • 路由结果增加审计日志，让 root 可以查看“为什么这条消息触发了工具”。
  • Agent 工具协议可以标准化成 tool schema，方便以后接搜索、日历、邮件、文件系统。
最需要小心的点：代码执行和主动群聊是两个高风险能力。前者要坚持确认和沙箱，后者要坚持白名单、冷却和 root 可控，否则很容易误触发或打扰群友。
